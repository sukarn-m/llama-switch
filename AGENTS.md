# AGENTS.md

> **⚠ Project-specific configuration.** The runtime `config/config.yaml` (gitignored) contains machine-specific paths, GPU topology, and bind addresses. Do not assume any example values — always review `config/config.yaml` for your own deployment.

Guidance for AI coding agents working in the `llama-switch` codebase.

## What this project is

`llama-switch` is a **GPU model multiplexer for `llama-server`** (from llama.cpp). It is an HTTP reverse proxy that presents a single OpenAI-compatible endpoint and transparently loads/unloads different GGUF models on one or more GPUs behind the scenes. It was written to replace a shell script (`llama-server.sh`) that ran a single fixed model — now many models can coexist and be switched on demand within VRAM and concurrency limits.

Key idea: a client sends `POST /v1/chat/completions` with `{"model": "my-model", ...}`. The proxy parses the `model` field, ensures that model's `llama-server` backend is running (spawning it if needed, evicting least-recently-used models to make room), and forwards the request — including SSE streaming — to that backend on a dynamically-allocated loopback port.

It is a **single-binary Go program** with no subpackages and one third-party dependency (`gopkg.in/yaml.v3`). Target environment is NVIDIA CUDA multi-GPU Linux (it shells out to `nvidia-smi` for VRAM accounting).

## Repository layout

The repo is split into three directories plus module/docs files at the root. All Go code is one `package main` in `src/` (no subpackages or internal packages).

| Path | Responsibility |

| `src/config.go` | YAML config types (`Config`, `ServerConfig`, `BackendConfig`, `ModelConfig`), loading, defaults, `~`/`$VAR` path expansion, llama-server argument & env building. |

## Architecture & request flow

```
Client ──HTTP──▶ ProxyServer (:8080)
                   │
                   ├─ GET  /                → health
                   ├─ GET  /v1/models       → OpenAI model list (with status field)
                   ├─ GET  /models          → llama.cpp router model list (with status)
                   ├─ POST /models/load     → explicitly load a model {"model":"id"}
                   ├─ POST /models/unload   → explicitly unload {"model":"id"} (Open WebUI "Eject")
                   ├─ GET  /v1/loaded       → legacy: list loaded models only
                   ├─ POST /admin/profile   → profile VRAM via running service (SSE stream)
                   └─ everything else       → handleProxy
                                            │
                    parse "model" from JSON body
                                            ▼
                   BackendManager.EnsureLoaded(modelID)
                     ├─ already loaded? → reuse
                     ├─ else: ensureCapacityLocked
                     │    ├─ enforce MaxModels (evict LRU)
                     │    └─ enforce per-GPU VRAM + 1GB headroom on GPU0 (evict LRU)
                     ├─ startBackendLocked: alloc port, spawn backend,
                     │    wait for health 200 (HealthTimeoutSeconds)
                     └─ return *Backend
                                            ▼
                   NewModelHandler(modelID) registered?
                     ├─ yes → forwardWithHandler: force non-streaming,
                     │        fetch response, ProcessResponse(), return JSON
                     └─ no  → forward(): proxy to 127.0.0.1:<port>,
                              stream response back if "stream":true
```

- Each loaded model = one `llama-server` (or custom binary) child process on its own port from `backend_port_start`–`backend_port_end`.
- Backends are addressed only on loopback (`127.0.0.1`); the public listener binds to the configured `host` (set in `config.yaml`).
- The idle sweeper goroutine (`idleSweeper`) unloads any backend untouched for `idle_timeout_minutes`, polled every `sweep_interval_seconds`.

## Per-model runtime override

Models can use a different binary than `backend.binary`, with their own arguments, environment, and health endpoint. This allows running non-llama-server backends (e.g. vLLM, Python servers) alongside regular GGUF models. Set these optional fields in a model's config:

| Field | Purpose | Default |
|-------|---------|---------|
| `binary` | Per-model binary path (overrides `backend.binary`) | `backend.binary` |
| `args` | Raw args list with `{port}` placeholder, bypasses llama-server flag construction | `BuildArgs()` output |
| `health_path` | Custom health check endpoint | `/health` |
| `env` | Per-model env vars, merged on top of `backend.env` | `backend.env` only |

When `args` is set, `BuildArgs()` uses the raw list directly (with `{port}` replaced by the dynamically assigned port) — no `-m`/`--mmproj`/`-c` flags are generated. When `binary` is set, it's resolved via PATH lookup. The rest of llama-switch (VRAM management, LRU eviction, idle sweeping, proxy forwarding) works unchanged because it's all process-and-HTTP-based.

## Model handlers

A model handler is a Go type implementing the `ModelHandler` interface (`MatchesModel`, `ProcessResponse`). When a request targets a model with a registered handler, `handleProxy` routes through `forwardWithHandler` instead of `forward`: the response is read in full, post-processed by the handler, and returned as a single JSON response. Streaming requests are forced to non-streaming (the handler needs the complete output).

Handlers are registered in `NewModelHandler()` in `src/chandra_handler.go`. To add a new handler: implement the interface in a new file under `src/`, add a registration line. To remove a handler: delete its file and the registration line. The generic infrastructure (`ModelHandler` interface, `forwardWithHandler`, `forceNonStream`) stays in `src/proxy.go` and can be reused by future handlers.

### Chandra 2 OCR handler

The Chandra handler (`src/chandra_handler.go`) post-processes responses from the Chandra 2 OCR model (a Qwen3.5-based VLM):

1. **Extracts HTML from the correct field** — checks `content` first, falls back to `reasoning_content` (Qwen3.5 thinking mode sometimes routes output there)
2. **Strips chain-of-thought** — removes any thinking text before the first `<div>` tag
3. **Converts structured HTML to markdown** — Chandra outputs `<div data-bbox="..." data-label="...">` blocks; the handler converts tables, lists, bold, underline, headers, and image descriptions to clean markdown
4. **Fixes Devanagari conjunct errors** — Chandra drops the `र` in certain conjuncts (कय→क्रय, केता→क्रेता, विकय→विक्रय, केती→क्रेती)
5. **Strips page headers/footers** — blocks with `data-label="Page-Header"` or `"Page-Footer"` are omitted

## Configuration reference (`config/config.yaml`)

- `server`: `host`, `port`, `backend_port_start/end`, `max_models`, `idle_timeout_minutes`, `health_timeout_seconds`, `sweep_interval_seconds`.
- `backend`: `binary` (path to `llama-server`), `env` (notably `LD_LIBRARY_PATH` for CUDA libs), `common_args` (flags prepended to every model invocation), `workdir`, `nvidia_smi`.
- `models`: list of `ModelConfig`. Lookup by `id` or `name` (`FindModel` matches `id` exactly, `name` case-insensitively). Each builds its llama-server args via `ModelConfig.BuildArgs`. Supports `mmproj` (multimodal), `tensor_split`, `ctx_checkpoints`, and freeform `extra_args`. Optional per-model override fields (`binary`, `args`, `health_path`, `env`) allow running non-llama-server backends — see [Per-model runtime override](#per-model-runtime-override).
- Path expansion: `~` and `$VAR` are expanded in config values (`expand()`), including colon-separated `LD_LIBRARY_PATH`-style lists.

**Defaults applied when fields are zero/empty:** `backend_port_start=8201`, `backend_port_end=8299`, `max_models=2`, `idle_timeout_minutes=60`, `health_timeout_seconds=180`, `sweep_interval_seconds=15`, `nvidia_smi="nvidia-smi"`.

## Commands

```
./bin/llama-switch serve [config]    # start proxy (default if no command given)
./bin/llama-switch profile [config]  # load each model once, measure VRAM, save to config/vram-cache.json
./bin/llama-switch models [config]   # list configured models + VRAM estimates
./bin/llama-switch status [config]   # query a running server's /v1/loaded  (note: wired in usage but dispatched via `serve` path; see main.go)
./bin/llama-switch help
```

Config path resolution: positional arg → `$LLAMA_SWITCH_CONFIG` → `config.yaml` next to the binary (`./` and `../config/` relative to it) → `~/.config/llama-switch/config.yaml` → `./config/config.yaml`.

## Conventions

- **Single package in `src/`.** All Go code is `package main` in `src/`. Do not introduce subpackages without reason; the project's style is one file per concern. Config files live in `config/`, the built binary in `bin/`.
- **Go 1.26** toolchain; only stdlib + `yaml.v3`. Avoid adding dependencies for things the stdlib can do.
- **Comment style:** section dividers use `// ── Name ──────` banners; exported functions carry doc comments; `Caller must hold <mu>` notes annotate lock contracts — respect these when editing.
- **Locking:** `BackendManager.mu` guards the `backends` map and port allocator interactions during lifecycle. Methods suffixed `Locked` require the caller to hold the lock (e.g. `startBackendLocked`, `stopLocked`, `ensureCapacityLocked`, `evictLRULocked`). Preserve the double-checked-locking pattern in `EnsureLoaded` (RLock fast path → Lock slow path).
- **Process output** from each backend is piped through `logPipe` and printed to stdout with a `[model-id]` prefix — keep this; it's how service logs are read.
- **Streaming detection** in `forward()` is a naive string scan for `"stream":true` / `"stream": true`. If you touch request handling, be aware this is intentionally simple and does not fully parse JSON.
- **Errors during backend start** (port alloc fail, binary missing, health timeout) must release the port and clean up — mirror the cleanup in `startBackendLocked`.
- **Config is authoritative.** `config/config.yaml` reflects a real deployment with specific GGUF paths, GPU topology, and a bind address. Treat it as live configuration, not an example template.
- **Tests exist** in `src/` (per-model runtime override in `llama_switch_test.go`, Chandra OCR handler in `chandra_handler_test.go`, plus `http_proxy_test.go`, `inflight_test.go`, `findmodel_vram_test.go`). Run `go test ./...` to execute all tests; use `-short` to skip integration tests.

## Common tasks for agents

- **Add a new model:** add an entry under `models:` in `config/config.yaml`; run `./bin/llama-switch profile` so admission control has VRAM estimates.
- **Add a non-llama-server model:** set `binary`, `args` (with `{port}`), and optionally `health_path`/`env` on the model config. The rest of llama-switch handles it automatically.
- **Add a new HTTP endpoint:** add a path branch in `ProxyServer.handle` (`src/proxy.go`) and a `handleXxx` method. Management/model-list endpoints return JSON; proxying goes through `handleProxy` → `forward`.
- **Add a model handler:** implement the `ModelHandler` interface in a new `*_handler.go` file under `src/`, register it in `NewModelHandler()` (currently in `src/chandra_handler.go`). The handler intercepts responses in `forwardWithHandler`.
- **Remove a model handler:** delete its `*_handler.go` and `*_handler_test.go` files from `src/`, remove the registration line in `NewModelHandler()`. The generic infrastructure in `src/proxy.go` can stay.
- **Change eviction policy:** `ensureCapacityLocked` and `evictLRULocked` in `src/backend.go`. LRU is based on `Backend.lastReq`, updated via `Touch()` on every proxied request.
- **Change CLI surface:** `src/main.go` `switch cmd` block + `usageText`. `parseConfigArg` handles config-path discovery.
- **Rebuild after edits:** `go build -o bin/llama-switch ./src` (replaces `bin/llama-switch`); restart the systemd unit if running as a service.

## Things to watch out for

- `ReadTimeout`/`WriteTimeout` on the proxy server are intentionally `0` to allow long-lived streaming requests — do not set finite timeouts without accounting for streaming.
- The VRAM headroom constant is `1024` MB (1 GB) in `ensureCapacityLocked` (`src/backend.go`). It is applied **only to the primary GPU** (CUDA0 / index 0) to account for OS/display-server overhead; secondary GPUs have no display output and thus no headroom. Adjust there if changing the safety margin.
- Per-GPU admission control (`vramFitsPerGPU` in `src/vram.go`) checks each GPU the model targets individually. If per-GPU profile data is unavailable (legacy cache entries without `gpu_vram`), it falls back to the aggregate VRAM check. Keep this fallback intact.
- VRAM measurements are validated during profiling (`validateVRAMMeasurement` in `src/vram.go`): rejected if below 256 MB or below the model's `.gguf` file size. Both the `profile` command and auto-profiling during `serve` enforce this. Rejected measurements are not cached.
- `waitHealth` polls every 2 s; large models may take close to `health_timeout_seconds` (default 180 s) to become ready. The proxy returns `503` with `Retry-After: 30` if a model fails to load.
- `nvidia-smi` is required for VRAM-aware scheduling; if it is unavailable, `ensureCapacityLocked` falls through optimistically (loads without VRAM checks) — keep this fallback intact.
- `config/config.yaml` contains absolute paths specific to the deployment machine; don't generalize them.
- In-flight request tracking: each `Backend` has an atomic `inFlight` counter, incremented in `handleProxy` before forwarding and decremented via `defer`. Eviction (`evictLRULocked`), idle sweeping (`sweepIdle`), and explicit unload (`StopModel`) all skip backends with `InFlight() > 0` to avoid killing active requests.
- Capacity queue: when `EnsureLoaded` can't evict (all backends in-flight), the caller releases the write lock and blocks on a channel (`capCh`) until a request completes and calls `SignalCapacity()`, or until the queue timeout (default 600s / 10 min). Requests for already-loaded models hit the fast path and never wait. Max queue depth is `queue_max_depth` (default 64); beyond that, new requests get an error. The queue is best-effort FIFO (goroutine scheduling order, not guaranteed).
- Drain mode: `BackendManager.StartDraining()` sets an atomic flag that causes `EnsureLoaded` and `handleProxy` to return 503. Used during remote profiling. `WaitIdle(timeout)` blocks until all in-flight counters reach zero.
- Remote profiling: `POST /admin/profile` (or `?force=true`) triggers profiling on the running service. The CLI `profile` command auto-detects a running service and delegates to this endpoint, streaming SSE output to stdout. Concurrent profiling requests are rejected.
- The `loading` map registration in `EnsureLoaded` happens AFTER the capacity wait loop, not before. If it were registered first, `len(loading)` would count against `MaxModels` in `ensureCapacityLocked`, causing a deadlock when two goroutines wait for different models simultaneously.
- `ensureCapacityLocked` adds +1 to `len(backends)+len(loading)` to account for the model being loaded (which hasn't been registered in `loading` yet).
